# 动态分组异步选股 - 设计方案

**日期**：2026-05-09
**适用页面**：动态分组创建页（解析后跳保存页）
**背景**：选股解析能力升级，耗时从原来的 1-3 秒变为不确定（高峰期可能 10s-1分钟），需要在不损失短耗时场景转化率的前提下，支持长耗时场景的异步交互。

---

## 一、目标与约束

### 目标
1. **短耗时场景（<10s）**：用户感知不到"异步"，留在创建页直到结果出来，最大化"创建 → 保存"转化
2. **长耗时场景（10s+）**：允许用户离开做别的事，结果完成后通过顶部 toast 召回到保存页
3. **失败容错**：服务端临时问题不应让用户重新填条件

### 关键约束
- 不允许使用全屏 loading（影响性能感知）
- 所有等待态提示必须 inline，不阻塞用户在创建页的浏览操作
- 文案需透露"高峰拥挤"等客观原因，降低用户烦躁感
- 不支持 App 杀进程后召回（不引入 push）

---

## 二、整体流程

```
[创建页] 提交条件
    ↓
[阶段 A] 0-3s：「查看结果」按钮内 loading（同现状）
    ↓ 超过 3s
[阶段 B] 3-10s：按钮下出现细进度条 + 客观文案
    ↓ 超过 10s
[阶段 C] 10s+：增加"先去看看其他"链接，允许后台化
    ↓
用户离开（任何阶段都可主动后台化或被动离开）
    ↓
[后台任务] 继续跑，结果缓存 1 小时
    ↓
完成 → 顶部 Toast 召回 → 跳转保存页
失败 → 重试 3 次（指数退避）→ 仍失败则回创建页让用户改条件
```

---

## 三、三阶段等待态设计

### 阶段 A：0-3s
- **视觉**：保持现状，仅"查看结果"按钮内部显示 loading 旋转图标
- **目的**：覆盖大多数非高峰时段（约 1-3s 完成的请求），用户完全感知不到异步逻辑
- **可操作性**：用户可在页面内自由滚动浏览已填条件，但条件项进入"只读"态（任务提交后不可修改条件，避免歧义）

### 阶段 B：3-10s
- **新增视觉元素**：
  - 「查看结果」按钮下方出现一条 2px 细进度条（真实进度，由服务端回报）
  - 进度条上方一行小字（次要文字色）：**"当前请求较多，正在为你筛选…"**
- **目的**：让用户知道"在做事"且"不是 App 卡了"，给客观原因解释
- **可操作性**：仍不阻塞操作，用户可以等也可以系统返回

### 阶段 C：10s+
- **新增视觉元素**：
  - 进度条上方文案变为：**"高峰时段处理较慢，"** + 紧跟一个**文字链接**："先去看看其他 →"
  - 进度条继续真实显示进度
- **目的**：主动给用户一个体面的离开方式，防止焦虑
- **可操作性**：
  - 不点链接 → 继续等待，进度条持续更新
  - 点链接 → 任务后台化，跳回上一级（自选页或之前的页面）

### 后台化触发方式
以下任一动作均触发"后台化"：
1. 用户在 C 阶段点击"先去看看其他 →"
2. 用户在任何阶段按系统返回键
3. 用户在任何阶段切到其他页（如点抽屉里的别的分组）

后台化 = 任务在服务端继续运行，前端记录 `task_id`，待完成时全局 toast 召回。

---

## 四、跨页 Toast 召回

### 触发条件
任务在后台完成、且用户当前不在保存页（在保存页则直接刷数据）。

### Toast 样式
```
┌────────────────────────────────────┐
│ ✓ 选股完成，共筛出 24 只  查看结果 │
└────────────────────────────────────┘
```
- 从屏幕顶部下滑进入
- 5 秒后自动收起
- 收起后**不留任何残留**（如红点、徽标等），保持自选页清爽
- 用户在 5 秒内点击 → 跳转到保存页（带 task_id 拉缓存结果）
- 5 秒过后用户错过了，下一次进入创建页时按"二次进入逻辑"处理（见下文）

**说明**：之前讨论过的"小红点常驻"方案被放弃，原因是开发成本较高、对自选页清爽度有影响。

---

## 五、二次进入创建页的处理

当用户后台化任务后，再次进入动态分组创建页：
- **创建页保持上次的条件**（不清空）
- 顶部展示一个状态条，显示当前任务状态：
  - 任务仍在进行：**"上次创建仍在进行中（进度 65%）"** + 进度条 + "等待结果"按钮
  - 任务已完成：**"上次创建已完成"** + "查看结果"按钮
  - 任务失败：**"上次创建失败，请调整条件后重试"**
- 用户可以选择等待原任务、查看已完成结果，或修改条件后**重新提交**（重新提交会取消旧任务）

---

## 六、失败与重试

### 重试策略
任务遇到服务端错误时，**自动重试 3 次**，间隔采用**指数退避**：
- 第 1 次重试：等待 1 秒
- 第 2 次重试：等待 2 秒
- 第 3 次重试：等待 4 秒
- 最坏情况下用户等待时间 = 原解析时间 + 7 秒

### 重试期间用户感知
- 重试在后台静默进行，**用户不感知**
- 进度条**冻结在重试发起时的位置**（不前进、不回退），文案不变
- 这样设计的目的：避免每次重试都打扰用户，绝大多数瞬时失败可以无感恢复
- 如果用户已经后台化离开，重试期间不弹任何 toast，等最终成功才召回，最终失败按下文规则处理

### 重试全部失败后
- 跳回创建页（如用户已离开则在其下次进入创建页时呈现该状态）
- 顶部展示提示条：**"网络繁忙，请稍后重试或调整条件"**
- 用户可直接再次提交，或调整条件重试
- 不限制连续失败后的再提交次数（避免误伤）
- 如用户在重试期间已离开到其他页（例如自选页），最终失败时**不弹 toast 打扰**，仅在其再次进入创建页时呈现失败状态（避免用户在看行情时被无关错误打断）

---

## 七、并发任务处理

同一用户**同时只允许 1 个 pending 任务**：
- 若已有选股任务还在跑，不允许新建或重新提交选股任务
- 创建页保持当前任务状态，只提供等待结果或离开后台化
- 任务完成、失败或超时失效后，用户才可重新提交

---

## 八、技术架构

### 任务状态机
```
pending（已提交，未开始）
    ↓
running（进行中，回报进度）
    ↓
success（完成，结果缓存 1 小时）
failed（失败，3 次重试用尽）
```

### 接口设计（建议）

| 接口 | 用途 |
|---|---|
| `POST /select-stocks/tasks` | 创建选股任务，返回 `task_id` |
| `GET /select-stocks/tasks/{id}` | 查询任务状态、进度、结果 |
| `POST /select-stocks/tasks/{id}/cancel` | 取消任务（重新提交时调用） |
| `GET /select-stocks/tasks/active` | 获取用户当前 pending/running 任务（用于二次进入） |

### 前端轮询
- 提交后立即开始轮询，**间隔 1.5s**
- 阶段 A/B/C 共用同一个轮询逻辑，UI 根据已等待时长切换样式
- 后台化后，由全局 toast 服务接管轮询（用户切到任意页都继续轮询）

### 进度条数据来源
后端在解析过程中按阶段更新进度：
- 解析条件完成：20%
- 拉取候选股票池：50%
- 应用筛选规则：80%
- 排序+组装结果：95%
- 完成：100%

具体百分比可由后端根据实际耗时占比微调。

### 缓存策略
- 任务结果在服务端缓存 **1 小时**
- 用户通过 toast 召回或二次进入查看时，直接读缓存，不重新解析
- **保存分组时**：使用 task_id 拉缓存的条件 + 重新拉一次最新行情数据组装结果（因为动态分组本身是条件式的，股价等数据每次访问都应是最新）

## 九、组件清单（前端）

| 组件 | 说明 |
|---|---|
| `SelectStockProgress` | 创建页底部的等待状态组件，含进度条 + 三阶段文案切换 |
| `BackgroundTaskService` | 全局任务管理服务，负责轮询、状态机、toast 触发 |
| `TaskCompleteToast` | 顶部下滑 toast 组件，5s 自动收起 |
| `ResumeTaskBanner` | 二次进入创建页时顶部的状态条 |

## 十、文案汇总

| 场景 | 文案 |
|---|---|
| 阶段 B 提示 | 当前请求较多，正在为你筛选… |
| 阶段 C 提示 | 高峰时段处理较慢，先去看看其他 → |
| Toast 成功 | ✓ 选股完成，共筛出 N 只  查看结果 |
| 二次进入-进行中 | 上次创建仍在进行中（进度 N%） |
| 二次进入-已完成 | 上次创建已完成 |
| 二次进入-失败 | 上次创建失败，请调整条件后重试 |
| 重试全失败 | 网络繁忙，请稍后重试或调整条件 |

---

## 十一、不在本期范围

- App 杀进程后的推送召回（需引入 push 通道，成本高）
- 多设备间任务同步（同一账号在另一设备上看到任务结果）
- 任务历史记录（用户查看过去几次的解析结果）
- 进度条的子步骤展示（如"正在筛选…"具体到哪一步）
