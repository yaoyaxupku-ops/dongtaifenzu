# 动态分组异步选股 - 实施计划

> **说明**：本计划是面向开发团队的**工程任务清单**，不包含代码级实现细节（因产品侧无代码库访问权）。每个任务包含：目标、关键决策、依赖、验收标准。开发团队在自己的代码库中落地时，可基于此清单进一步拆分为 TDD 粒度的子任务。

**关联设计文档**：[2026-05-09-动态分组异步选股-design.md](../specs/2026-05-09-动态分组异步选股-design.md)

**目标**：支持选股解析的异步交互，兼顾短耗时场景的高转化和长耗时场景的不流失

**总体架构**：前端轮询 + 服务端任务队列 + 全局任务管理服务 + 跨页 Toast 召回

**技术栈假设**：移动端 App（iOS/Android），RESTful API，不引入 WebSocket / Push

---

## 阶段划分

```
M1: 后端任务化改造（2周）        ←── 必须先完成，前端才能开发
M2: 前端异步交互核心（2周）      ←── M1 完成后可并行多端
M3: 体验增强与异常处理（1周）    ←── 重试、覆盖、二次进入
M4: 灰度与监控（1周）            ←── 上线保障
```

---

## M1：后端任务化改造

### T1.1 任务表与状态机

**目标**：将原同步选股接口改造为任务化模型

**关键决策**：
- 任务表字段：`task_id`、`user_id`、`condition_json`（用户填的条件）、`status`（pending/running/success/failed/cancelled）、`progress`（0-100）、`result_cache_key`、`error_code`、`retry_count`、`created_at`、`started_at`、`completed_at`、`expire_at`
- `task_id` 用 UUID 或雪花 ID，不依赖自增
- `status` 转移仅允许单向（pending → running → success/failed/cancelled），用 DB 乐观锁防止并发写坏
- `expire_at` = `completed_at + 1 小时`（缓存有效期）

**依赖**：无

**验收**：
- 任务表建立，能正确记录一次选股的完整生命周期
- 状态非法转移（如 success → running）被拒绝并记日志
- 对 `user_id + status in (pending, running)` 建索引，支持"查当前活跃任务"

---

### T1.2 提交任务接口

**目标**：`POST /select-stocks/tasks`

**关键决策**：
- 入参：条件 JSON（与现有同步接口保持一致，降低改造成本）
- 出参：`{ task_id, status: "pending", estimated_duration }`
- **并发控制**：若该用户已存在 pending/running 任务，**不自动覆盖**，返回特定错误码 `TASK_IN_PROGRESS` 和旧 `task_id`，由前端弹"覆盖确认"
- 提交后立即入队，不同步等待

**依赖**：T1.1

**验收**：
- 正常提交返回 `task_id`
- 同一用户重复提交返回 `TASK_IN_PROGRESS` + 旧 `task_id`
- 接口 P99 响应时间 < 200ms（不等选股完成）

---

### T1.3 任务状态查询接口

**目标**：`GET /select-stocks/tasks/{task_id}`

**关键决策**：
- 出参：`{ task_id, status, progress, result?, error_code?, expire_at }`
- `result` 仅在 `status=success` 时返回，且格式与原同步接口返回一致（前端保存页无需改造取数逻辑）
- 鉴权：只能查自己的任务

**依赖**：T1.1

**验收**：
- 各状态返回字段正确
- 跨用户查询返回 403
- 接口 P99 < 100ms（支持 1.5s 轮询，需要轻）

---

### T1.4 取消任务接口

**目标**：`POST /select-stocks/tasks/{task_id}/cancel`

**关键决策**：
- 仅允许取消 pending/running 状态的任务
- 已完成的任务调用此接口返回成功但不改状态（幂等）
- 取消后 worker 若还在跑，检测到状态变更后提前退出

**依赖**：T1.1

**验收**：
- 能正确取消活跃任务
- 对已完成任务幂等
- 覆盖场景下：前端"覆盖确认"点击后，先调用此接口，再调用提交接口

---

### T1.5 查询活跃任务接口

**目标**：`GET /select-stocks/tasks/active`

**关键决策**：
- 返回该用户当前 pending/running/已完成未过期的**最近一条**任务
- 供前端"二次进入创建页"时查询是否有进行中或刚完成的任务
- 出参同 T1.3，增加一个 `found: bool` 字段

**依赖**：T1.1

**验收**：
- 无活跃任务时返回 `found=false`
- 有进行中任务返回完整状态
- 刚完成但在 1 小时缓存期内的任务也能返回

---

### T1.6 解析任务 Worker

**目标**：后台 worker 消费任务队列，执行选股解析

**关键决策**：
- 每执行到一个里程碑，更新 `progress` 字段（20/50/80/95/100，具体节点按实际耗时占比微调）
- 进度更新频率：不超过 1 秒 1 次（避免 DB 压力）
- Worker 内部失败时**不直接标 failed**，而是等 T1.7 的重试逻辑决定
- 支持水平扩展（多实例消费，通过 DB 行锁或队列本身的 visibility timeout 保证不重复执行）

**依赖**：T1.1

**验收**：
- 正常解析能推进 progress 字段
- 任务被取消时 worker 提前退出
- 高峰期多实例并发消费不冲突

---

### T1.7 失败重试机制

**目标**：选股失败时自动重试 3 次，指数退避

**关键决策**：
- 重试间隔：1s / 2s / 4s
- 只重试**服务端错误**（5xx、超时、下游依赖抖动），**不重试**客户端错误（条件非法、用户数据缺失等 4xx 类）
- 重试期间 `status` 保持 `running`，`progress` 保持不变
- `retry_count` 字段记录已重试次数
- 3 次都失败，`status=failed`，写入 `error_code`

**依赖**：T1.6

**验收**：
- 模拟 3 次失败后第 4 次成功 → 最终 status=success
- 模拟 4 次失败 → status=failed，retry_count=3
- 客户端错误立即失败，不重试

---

### T1.8 结果缓存与过期清理

**目标**：成功的任务结果缓存 1 小时，过期后清理

**关键决策**：
- 结果存 Redis 或同类 KV，key 格式：`select_stocks:result:{task_id}`
- TTL = 1 小时
- 定时任务（每 5 分钟）清理 `status=success/failed/cancelled` 且 `expire_at < now` 的任务记录（也可用 DB 的 TTL 能力）
- 保留失败任务记录供分析，但 `error_code` 等字段保留，不清 result（failed 本身就没 result）

**依赖**：T1.6

**验收**：
- 成功 59 分钟后仍能拉到结果
- 61 分钟后拉结果返回过期错误
- 过期记录被清理，不积压

## M2：前端异步交互核心

### T2.1 全局任务管理服务

**目标**：前端建立一个单例服务，负责任务生命周期管理

**关键决策**：
- **职责**：
  1. 保存当前活跃 `task_id`（内存 + 本地持久化，防杀 App 丢失）
  2. 轮询接口（间隔 1.5s）
  3. 状态变化广播（事件总线 / observable）
  4. Toast 触发
- **持久化**：使用 App 的本地存储（iOS: UserDefaults, Android: SharedPreferences, 或 RN: AsyncStorage）
- **生命周期**：App 启动时若本地有未完成的 task_id，立即调用 T1.5 恢复；完成后清本地存储
- **前台/后台**：App 切到后台时轮询暂停，切回前台立即发一次查询

**依赖**：T1.3, T1.5

**验收**：
- 切页面不打断轮询
- App 后台→前台能恢复查询
- 任务完成/失败时能广播给当前页面

---

### T2.2 创建页等待态组件（三阶段）

**目标**：创建页下方的等待态 UI，按等待时长切换形态

**关键决策**：
- **阶段 A（0-3s）**：仅「查看结果」按钮内显示 loading，条件区进入只读态（置灰或加锁定图标）
- **阶段 B（3-10s）**：按钮下方出现 2px 细进度条 + 文案"当前请求较多，正在为你筛选…"
- **阶段 C（10s+）**：文案变为"高峰时段处理较慢，" + 文字链接"先去看看其他 →"
- 阶段切换基于**客户端计时**（从提交时开始计），不依赖服务端
- 进度条数据来自 T1.3 的 `progress` 字段
- 用户点文字链接或按返回键 = 触发后台化（任务保留，页面返回上一级）

**依赖**：T2.1, T1.3

**验收**：
- 3s 阈值精准切换
- 10s 阈值精准切换
- 进度条跟随服务端 progress 平滑更新
- 点链接能正确后台化并返回

---

### T2.3 Toast 召回组件

**目标**：全局顶部 Toast，任务完成时通知用户

**关键决策**：
- 样式：顶部下滑进入，5 秒自动收起
- 内容：`✓ 选股完成，共筛出 {N} 只  查看结果`
- 全局挂载（在根导航上），覆盖除保存页外的所有页面
- **不显示**场景：用户当前在保存页（直接刷新数据即可）、用户当前在该任务的创建页等待态中（不需要 toast）
- 点击跳转保存页时携带 `task_id`，保存页用此 id 从 T1.3 拉缓存结果

**依赖**：T2.1, 保存页改造（T2.5）

**验收**：
- 在自选页收到 toast 并点击能跳到保存页
- 在其他任意页（行情、我的、搜索等）都能显示
- 5 秒自动消失，消失后无残留
- 用户在保存页不出现 toast

---

### T2.4 创建页：任务提交与触发流程

**目标**：改造现有"查看结果"按钮的点击逻辑

**关键决策**：
- 原逻辑：同步调用选股接口 → 拿到结果 → 跳保存页
- 新逻辑：
  1. 调用 T1.2 提交任务 → 拿到 task_id
  2. 通知 T2.1 全局服务开始跟踪此任务
  3. 进入阶段 A 等待态
  4. 如果 3s 内收到 success → 直接跳保存页（用户体验同原来）
  5. 如果超时进入 B/C 阶段，由用户决定留下等还是后台化
- 任务完成时（无论在哪个阶段），自动跳保存页（前提是用户还在创建页）

**依赖**：T1.2, T2.1, T2.2

**验收**：
- 1-2s 返回的短耗时场景：用户感知与原来几乎一致
- 15s 的长耗时场景：用户可以留下等，也可以后台化
- 后台化后任务完成能正确触发 toast

---

### T2.5 保存页：支持从缓存读取

**目标**：保存页改造为优先从 task_id 读取缓存结果

**关键决策**：
- 入参：`task_id`（新增，原来可能是直接传 result 或 condition）
- 页面加载时调用 T1.3 拉结果
- 状态：loading / success（渲染结果）/ expired（缓存过期，提示"结果已过期，请返回重新筛选"）
- **保存时**：后端保存分组条件，不保存这一刻的结果（动态分组本身是条件式的）

**依赖**：T1.3

**验收**：
- 从 toast 跳转能正确渲染结果
- 从创建页直接跳转（短耗时场景）也能正常渲染
- 缓存过期场景有明确提示和退路

## M3：体验增强与异常处理

### T3.1 覆盖确认弹窗

**目标**：并发任务场景的确认交互

**关键决策**：
- 触发时机：用户提交任务时，T1.2 返回 `TASK_IN_PROGRESS` 错误
- 弹窗文案：**"上次创建仍在进行中，是否覆盖？"**
- 按钮：「等等」（关闭弹窗，不做任何操作）/ 「覆盖」（先调 T1.4 取消，成功后再调 T1.2 提交）
- 覆盖失败的容错：取消接口调用失败时，提示"操作失败，请稍后重试"，不提交新任务

**依赖**：T1.2, T1.4

**验收**：
- 有活跃任务时提交触发弹窗
- 「覆盖」能正确取消旧任务并启动新任务
- 「等等」不影响旧任务继续执行

---

### T3.2 二次进入创建页的状态恢复

**目标**：后台化任务后，用户再次进入创建页时，展示上次任务状态

**关键决策**：
- 进入时调用 T1.5 查询活跃任务
- 三种场景：
  - **进行中**：顶部状态条"上次创建仍在进行中（进度 N%）" + 进度条 + 保留原条件（只读）
  - **已完成未过期**：顶部状态条"上次创建已完成" + 「查看结果」按钮（点击跳保存页）
  - **失败**：顶部状态条"上次创建失败，请调整条件后重试"（红色）+ 条件恢复为可编辑
  - **无活跃任务**：空白创建页（不做任何提示）
- 条件恢复：从 T1.3/T1.5 返回的 `condition_json` 还原 UI 状态

**依赖**：T1.5

**验收**：
- 三种状态切换正确
- 条件还原 100% 还原（所有字段）
- 无活跃任务时不打扰

---

### T3.3 重试失败后的用户提示

**目标**：3 次重试都失败后的用户引导

**关键决策**：
- 用户**在创建页等待态**时失败：停留在创建页，顶部出现红色提示条"网络繁忙，请稍后重试或调整条件"，条件区解锁可编辑，用户可直接再次提交
- 用户**已离开（后台化）**时失败：**不弹 toast 打扰**（避免用户在看行情时被无关错误打断），仅在其下次进入创建页时呈现失败状态（由 T3.2 处理）
- 失败后不限制用户再次提交的次数

**依赖**：T1.7, T3.2

**验收**：
- 等待态失败正确提示并解锁条件
- 后台化失败不弹 toast
- 下次进入创建页能看到失败状态

---

### T3.4 前端轮询的异常处理

**目标**：轮询过程中网络错误的容错

**关键决策**：
- 单次轮询失败：忽略，继续下一次（1.5s 后）
- 连续 10 次轮询失败（约 15s）：提示"网络不稳定"，但任务本身在服务端继续跑，用户拉动刷新或重新进入可恢复
- 完全离线：轮询暂停，网络恢复后自动恢复

**依赖**：T2.1

**验收**：
- 偶发网络错误不影响体验
- 长时间断网有明确提示
- 网络恢复后能自动接续

## M4：灰度与监控

### T4.1 埋点

**目标**：建立关键指标的埋点体系

**关键决策**：
- 必埋事件：
  - `select_stock.submit`：提交任务（携带条件复杂度等级）
  - `select_stock.stage_b_shown`：进入阶段 B
  - `select_stock.stage_c_shown`：进入阶段 C
  - `select_stock.background`：用户主动后台化（区分点击链接/返回键）
  - `select_stock.toast_shown`：toast 出现
  - `select_stock.toast_clicked`：toast 被点击
  - `select_stock.toast_ignored`：toast 5s 消失未被点
  - `select_stock.group_saved`：最终成功保存分组
  - `select_stock.fail`：3 次重试用尽
  - `select_stock.overwrite_confirm`：覆盖确认弹出
- 核心北极星指标：**"创建任务 → 保存分组"的漏斗转化率**（按耗时区间分桶看）

**依赖**：无

**验收**：
- 各事件能正确触发
- 数据看板能按耗时分桶统计转化率

---

### T4.2 服务端监控与告警

**目标**：核心指标监控

**关键决策**：
- 监控项：
  - 任务提交 QPS
  - 各状态任务数（pending/running 堆积）
  - P50/P95/P99 解析耗时
  - 失败率（按 error_code 分类）
  - 重试成功率
- 告警：pending 任务堆积 > 阈值（说明 worker 消费跟不上）、失败率突增、P99 耗时超 60s

**依赖**：T1.6

**验收**：
- 告警能在异常时触发
- 监控面板能快速定位问题

---

### T4.3 灰度发布

**目标**：分批放量，快速回滚能力

**关键决策**：
- 第一批：5% 用户，观察 3 天
- 第二批：30% 用户，观察 3 天
- 第三批：全量
- 灰度开关：服务端配置，前端通过开关判断走新异步逻辑还是旧同步逻辑
- **回滚预案**：开关关闭后，前端立即走旧同步接口，后端任务化接口继续存活但不被调用

**依赖**：全部前置任务

**验收**：
- 灰度开关能实时生效（无需发版）
- 回滚后存量任务不影响（已提交的任务能正常完成）
- 核心指标（转化率、失败率、P99 耗时）有明显看板跟踪

---

### T4.4 Bug Bash 与上线清单

**目标**：上线前集中测试覆盖关键路径

**关键决策**：
- 必测场景（按优先级）：
  1. 短耗时场景（2s 完成）：用户体验与原来一致
  2. 长耗时场景（30s 完成）：阶段 B/C 切换正常
  3. 后台化 → Toast 召回 → 跳保存页完整流程
  4. 二次进入创建页的三种状态恢复
  5. 覆盖确认弹窗
  6. 3 次重试全部失败的用户提示
  7. App 被切到后台再回前台
  8. 弱网环境（3s 请求延迟、丢包）
  9. 并发提交（快速点击"查看结果"按钮）
  10. 缓存过期（1 小时后访问 toast 链接）

**依赖**：全部前置任务

**验收**：
- 10 个场景全部通过
- 重大问题清零
